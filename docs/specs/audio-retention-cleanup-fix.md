# Audio Retention Cleanup — Honor the Configured Setting

## Status
Implemented

## Problem / Goal

`_setupAudioCleanup()` in [`src/helpers/ipcHandlers.js:734-753`](../../src/helpers/ipcHandlers.js#L734) hardcodes:

```
const DEFAULT_RETENTION_DAYS = 30;
```

and passes that constant, once at startup and again every 6 hours forever, to:

- `AudioStorageManager.cleanupExpiredAudio(retentionDays, databaseManager)` — [`src/helpers/audioStorage.js:102`](../../src/helpers/audioStorage.js#L102), deletes dictation `.webm` files (and clears their DB `has_audio` flags) whose mtime is older than `Date.now() - retentionDays * 86400000`.
- `meetingAudioStorage.cleanupExpiredAudio(retentionDays)` — [`src/helpers/meetingAudioStorage.js:156`](../../src/helpers/meetingAudioStorage.js#L156), same math, deletes meeting `.webm` files.

Meanwhile the user-facing control is `audioRetentionDays`, defined and persisted entirely in the renderer (`src/stores/settingsStore.ts`, `NUMERIC_SETTINGS`, default `0` when unset per lines 1065-1071; UI in `src/components/SettingsPage.tsx:2358-2382` offering `0/1/7/14/30/60/90` days). **This value is never sent to the main process.** No matter what the user picks, the main process silently deletes audio at a fixed 30-day mark, every 6 hours, forever. This is the same "two sources of truth" bug class already tracked in project memory: a setting lives in renderer `localStorage` and the main process has its own independent, hardcoded notion of it.

### Confirmed semantics of `audioRetentionDays = 0` (resolved with the user)

The UI currently labels `0` as "Disabled" with description copy implying "no automatic deletion." An earlier draft of this spec followed that reading (`0` = skip cleanup, keep audio forever). **This has been overridden by explicit user decision: `audioRetentionDays = 0` means "delete all existing audio / retain nothing," not "keep everything forever."** Concretely: on every cleanup tick, if the effective retention value is `0`, every existing dictation and meeting `.webm` file is deleted, regardless of age. This is the opposite of what the current UI copy says, and this spec's Design section below now includes bringing that copy in line with the new behavior (see Requirement 9 / Design → "UI copy").

One consequence worth calling out precisely: with this semantics, `0` no longer needs a special "skip deletion" branch — the existing naive cutoff formula (`Date.now() - retentionDays * 86400000`) already produces "delete everything" for `retentionDays = 0` on its own, since `cutoffMs` becomes "right now" and every pre-existing file's mtime is earlier than that instant. **No code branch is needed to make `0` mean "delete everything" — it falls out of the existing math for free.** What *does* need an explicit guard is the opposite failure mode: malformed input that is not `0` but also not a valid non-negative number (a negative value, `NaN`, `Infinity`, or non-numeric junk from a hand-edited `.env` file — never producible by the UI's fixed `0/1/7/14/30/60/90` dropdown, but possible via direct `.env` tampering or a future caller bypassing normalization). Two distinct silent failure modes are possible if this isn't guarded against:
- A negative `retentionDays` pushes the cutoff into the *future*, which — via the same unguarded formula — also happens to delete every existing file, but for the wrong reason (accidental, not the deliberate "0 means wipe" behavior).
- A non-finite `retentionDays` (e.g. `NaN` from a corrupted value) produces `cutoffMs = NaN`, and `mtimeMs < NaN` is always `false` in JavaScript — so cleanup silently does *nothing*, forever, with no error and no indication that the configured value was garbage.

Both are worth defending against explicitly, and by the same guard: only proceed with the delete pass when the value is a finite number `>= 0`; treat anything else (negative, `NaN`, `Infinity`, unparseable) as invalid and skip deletion for that tick, logging a warning. **Do not conflate "invalid input" with "the user chose 0"** — they must produce the same safe outcome (no deletion) for different, explicitly-documented reasons, not because invalid input happens to be coerced into looking like `0`.

### Fallback default when no value has ever synced from the renderer

**Final decision, confirmed by the user: the fallback default is `0`** (not 30, as an earlier draft of this spec recommended). This applies to any run where the main process has never received a synced value from the renderer (fresh install, first launch before the renderer's settings hydration completes, or a headless/CLI-bridge session that never opens a window). The user's stated rationale (translated): *"the fallback should be non-retain... zero... the system's premise is privacy."* This is a deliberate design stance, not a placeholder — the app defaults to privacy/no-retention rather than to convenience/legacy-30-day retention, and local audio storage is intentionally **opt-in**: a user must actively choose a non-zero retention window in Settings for any audio to persist past the next cleanup tick. This is final and is not to be reverted or second-guessed further by anyone implementing or reviewing this spec.

**This combination (fallback = 0, and 0 = delete-everything) has a materially aggressive emergent consequence, confirmed as intended and documented here for anyone implementing or reviewing this spec:** the renderer's own default-when-unset for `audioRetentionDays` (`settingsStore.ts` line 1065-1071) has *always* been `0` — this was already true before this bug fix, but it was inert, because the main process ignored the renderer's value entirely and always used the hardcoded 30-day window regardless. After this fix, that same "user never touched the dropdown" state (still `0` on both sides) will, for the first time, actually reach the main process — and will now mean "delete all audio," not "do nothing." **In other words: any user who installs the app, dictates something, and never opens Settings → Privacy & Data will have every locally-stored dictation/meeting audio file deleted starting from the first cleanup tick, with zero action on their part.** This is not merely a narrow first-run race condition — it is the default, out-of-the-box behavior for the common case of a user who never visits that settings page at all (confirmed: `OnboardingFlow.tsx` has no retention-related step, so onboarding does not set this value either). **The user was shown this exact consequence explicitly and confirmed it is intended: privacy-by-default over retention-by-default.** See Design for the narrow startup-ordering safeguard this spec still includes for the genuinely-never-synced/headless case (a real, independent data-loss-during-upgrade protection, not a hedge against the confirmed default itself).

### Investigation: does `dataRetentionEnabled` also need to feed into this cleanup job?

No. Confirmed by reading `src/helpers/audioManager.js` (`saveTranscription`/`saveFailedTranscription`, lines ~2504-2547) and `src/helpers/discardedRecording.js`:

- `dataRetentionEnabled` is a **write-time** master switch: when `false`, `audioManager.js` skips saving the transcription DB row (and its audio) in the first place. It is checked *before* a `.webm` file or DB row ever gets created.
- `audioRetentionDays` serves two purposes today: (a) a write-time prerequisite for the "save discarded recordings" feature (`shouldSaveDiscardedRecording()` in `discardedRecording.js` requires `audioRetentionDays > 0` before it will save a discarded recording's audio), and (b) the retention/cleanup window this bug is about.
- There is no code path where toggling `dataRetentionEnabled` off retroactively purges already-saved audio/history; it only gates future writes.

Conclusion (unaffected by the semantics inversion): `dataRetentionEnabled` and `audioRetentionDays` are orthogonal. `_setupAudioCleanup()` keys **only** off `audioRetentionDays`.

### Re-examined: does `shouldSaveDiscardedRecording()`'s existing `audioRetentionDays > 0` gate still make sense under the new semantics?

Yes — re-checked specifically because the semantics inverted. `src/helpers/discardedRecording.js` requires `settings.audioRetentionDays > 0` before it will save a discarded recording's audio. Under the *old* (pre-inversion) reading this gate was a little odd on reflection (it refused to save discarded audio specifically when retention was "disabled," i.e. when nothing would ever auto-delete it — arguably the *safest* case to keep it). Under the *new*, confirmed semantics, this gate makes clean, direct sense without any change: if `audioRetentionDays === 0`, any newly-saved discarded-recording audio file would simply be deleted again on the very next cleanup tick (since `0` now means "delete everything"), so refusing to bother writing it in the first place is the more correct behavior, not a coincidental one. **Conclusion: no change needed to `discardedRecording.js`.**

### Investigation: does meeting audio need its own retention setting?

No separate setting exists. Grepped `settingsStore.ts`, `SettingsPage.tsx`, `useSettings.ts` for `meetingRetention`/`meetingAudioRetention` — no matches. Today, `meetingAudioStorage.cleanupExpiredAudio()` is already called with the *same* hardcoded `DEFAULT_RETENTION_DAYS` constant as dictation audio. This spec preserves that coupling (both cleaned up under the same configured window) rather than introducing new UI surface for a second knob.

## Requirements

1. `_setupAudioCleanup()`'s recurring job (`runCleanup()`, invoked once immediately and then every 6 hours via `setInterval`) must read the user's *currently configured* retention value fresh on every invocation — not a value captured once at startup — since the user can change the setting without restarting the app.
2. A configured value of `0` must result in **all existing dictation and meeting audio files being deleted** on that cleanup tick, regardless of file age. This must not require a special-cased branch that treats `0` differently from other valid values — the existing cutoff formula already produces this outcome for `retentionDays = 0` (see Problem section). The only special-casing needed is the validity guard in Requirement 3.
3. A `retentionDays` value that is **not** a finite number `>= 0` (negative, `NaN`, `Infinity`, unparseable) must be treated as invalid and must result in the cleanup job skipping deletion entirely for that tick (logging a warning) — explicitly *not* conflated with the valid, deliberate value `0`, which deletes everything. This guard belongs both at the `ipcHandlers.js` caller level and, as defense-in-depth, inside `cleanupExpiredAudio()` in both `src/helpers/audioStorage.js` and `src/helpers/meetingAudioStorage.js`, so any other/future caller of these exported functions is protected too.
4. The main process needs a `PERSISTED_KEYS`-backed, `.env`-persisted setting (`AUDIO_RETENTION_DAYS`) that survives app restarts independently of whether the control panel window is open, mirroring the existing `EnvironmentManager` `get<Thing>()`/`save<Thing>()` pattern (e.g. `getUiLanguage`/`saveUiLanguage`, `getActivationMode`/`saveActivationMode` in `src/helpers/environment.js`).
5. The renderer must push `audioRetentionDays` changes to the main process via a dedicated IPC round trip when the user changes the setting, and must hydrate its own state from the main process's persisted value on startup (reconciling `localStorage`/Zustand state toward the main-process value), mirroring the existing `getActivationMode`/`getUiLanguage`/`getVoiceAgentKey` startup-sync block in `src/stores/settingsStore.ts` (~lines 2266-2310).
6. When the main process has never received any synced value at all (`AUDIO_RETENTION_DAYS` env var literally never written), the fallback default is `0` — **final decision, confirmed by the user** (deliberate privacy-by-default stance; local audio storage is opt-in, not opt-out; see Problem section for the exact rationale and the confirmed, intended consequence that new/never-configured installs delete all audio by default). A narrow startup-ordering safeguard (Design, below) distinguishes "genuinely never configured" from "explicitly set to 0" for the very first cleanup pass only, to avoid destroying data on a headless/no-window session (or an existing user's first post-upgrade launch, before their previously-chosen renderer value has a chance to sync) — see Design for exactly what this safeguard protects against and what it does not.
7. Meeting audio cleanup (`meetingAudioStorage.cleanupExpiredAudio`) shares the same `AUDIO_RETENTION_DAYS`-derived value as dictation audio cleanup (no separate meeting-specific setting is introduced by this fix).
8. A regression test must prove the *configured* (non-default) retention period is honored end-to-end (files older than a configured, non-30 cutoff are deleted, newer ones survive), must prove `0` deletes **all** files regardless of age (not just old ones), and must prove invalid/negative input deletes **nothing** (distinct from `0`) — in a way that would have caught today's bug (i.e., it must exercise the code path that decides *which* retention value to use, not just the already-correct low-level file-deletion function).
9. The UI copy describing this setting (`settingsPage.privacy.audioRetentionDescription` and `settingsPage.privacy.audioRetentionDisabled` in the translation files) must be updated to accurately reflect that the `0`/lowest option now means "delete all local audio immediately," not "no automatic deletion" — see Design → "UI copy" for scope and proposed wording. Shipping this fix with the current, now-backwards copy in place is not acceptable, since a user reading "Disabled" would reasonably expect "nothing happens to my audio," which is now the opposite of the truth.

## Non-goals

- Changing the 6-hour interval cadence itself (`SIX_HOURS_MS`) — out of scope; this fix is about *what value* each tick uses, not *when* ticks happen.
- Making a setting change trigger an immediate, out-of-cycle cleanup sweep — **confirmed by the user: wait for the next scheduled tick (or app restart), no immediate sweep on change.** Users who want existing audio gone right now can already use the "Clear All Audio" button, or set the dropdown to `0` and wait for the next tick.
- Adding a separate, dedicated retention control for meeting audio — no separate setting exists today, and none is introduced by this fix.
- Any change to `dataRetentionEnabled` or `saveDiscardedTranscriptions` semantics, or to `discardedRecording.js` — both re-confirmed orthogonal/correct under the new semantics (see Problem section) and left untouched.
- Piggybacking on the `sync-startup-preferences` IPC channel — that handler's scope is provider/model pre-warming with unrelated side effects; a small dedicated IPC pair is the better fit (see Design).
- Backfilling translations for the 7 additional languages CLAUDE.md claims exist (`es, fr, de, it, ru, zh-CN, zh-TW`) — only `src/locales/en/translation.json` and `src/locales/pt/translation.json` actually exist on disk today (confirmed via glob); this spec updates those two only. The CLAUDE.md/reality mismatch on language count is a pre-existing documentation gap, not something this fix introduces or is responsible for correcting.

## Design

### `src/helpers/environment.js`

- Add `"AUDIO_RETENTION_DAYS"` to the `PERSISTED_KEYS` array (near `PANEL_START_POSITION`/`START_MINIMIZED`/`UI_LANGUAGE`) so `_writeEnvFile()` actually persists it to `userData/.env`.
- Add `getAudioRetentionDays()`: reads the `AUDIO_RETENTION_DAYS` env var via `_getKey()`, parses it as an integer. If the raw value is missing, empty, or fails to parse as a finite non-negative integer, **fall back to `0`** — final, confirmed default (see Problem section for the user's privacy-by-default rationale and the intended, confirmed consequence this produces). This getter's job is to always return a clean, valid, non-negative integer — it must never emit a negative number or `NaN` to callers; any malformed persisted value normalizes to the `0` fallback the same as a genuinely-missing value (there is no meaningful distinction to preserve past this layer, since the fallback itself is now also `0`).
- Add `saveAudioRetentionDays(days)`: normalizes the incoming value to a non-negative integer (floor; coerce negative/non-finite input to `0`), stores it via `_saveKey("AUDIO_RETENTION_DAYS", ...)`, and — matching every other non-secret setter in this file — fire-and-forget calls `this.saveAllKeysToEnvFile().catch(() => {})` before returning `{ success: true, days }`.

### `src/helpers/ipcHandlers.js`

- Rewrite `_setupAudioCleanup()`'s `runCleanup()` closure to read `this.environmentManager.getAudioRetentionDays()` fresh on every call instead of closing over a hardcoded constant. Remove the `DEFAULT_RETENTION_DAYS` local constant entirely.
- Validity guard (Requirement 3): only call `this.audioStorageManager.cleanupExpiredAudio(...)` and `meetingAudioStorage.cleanupExpiredAudio(...)` when the resolved value is a finite number `>= 0`. Since `getAudioRetentionDays()` already guarantees this, this guard is defense-in-depth against the theoretical case of `environmentManager` being replaced/mocked or the getter's contract being violated in the future — it costs nothing and matches the same guard needed one layer down.
- **Startup-ordering safeguard for the never-synced case (Requirement 6) — corrected mechanics (post-review):** distinguish "the `AUDIO_RETENTION_DAYS` env var has never been written even once" from "it has been written and its value happens to be `0`," using `environmentManager.hasAudioRetentionDaysBeenSet()`. If the key has never been persisted, the *very first, immediate* `runCleanup()` invocation at app boot is skipped (falling through directly to starting the 6-hour `setInterval`), giving the renderer's startup sync (Requirement 5) a chance to establish the real, authoritative value before any file is ever touched.
  - **Critical correction: the main process must NOT self-persist a value here.** An earlier implementation had `_setupAudioCleanup()` write `saveAudioRetentionDays(getAudioRetentionDays())` (i.e. read the `0` fallback and write it straight back) when the key was unset, to mark it "set" so the skip only ever happens once. This is a real bug: `_setupAudioCleanup()` runs inside the `IPCHandlers` constructor, which `main.js` calls *before* any window (and therefore the renderer's `localStorage`) exists. Self-persisting `0` at that point permanently marks the key "set" with a bogus fallback *before* an existing user's real, previously-chosen renderer value (e.g. `30`, from before this fix ever shipped) has any chance to sync — and since the renderer's own startup sync treats a "set" main value as authoritative, it would then pull and overwrite the user's real `30` with `0`, then lose all their audio on the next tick. Deterministic, not a rare race: it fails for every pre-existing user with a real preference. **Fix: `_setupAudioCleanup()` only decides whether to skip the immediate pass (via a new pure helper, `shouldRunImmediateCleanup(hasBeenSetOnMain)` in `audioCleanupPolicy.js`) — it never writes `AUDIO_RETENTION_DAYS` itself.** Establishing the real, authoritative value for the first time is entirely the renderer startup sync's responsibility (see `src/stores/settingsStore.ts` below).
  - **What this safeguard protects, stated plainly:** it protects (a) the case where the app never gets a chance to sync at all (e.g. a headless/CLI-bridge-only session, or a crash before the first window loads), and (b) an *existing* user's very first launch after upgrading to this fix — their previously-chosen renderer value (e.g. `30`) needs a chance to sync into the brand-new `AUDIO_RETENTION_DAYS` env var before the first cleanup pass runs. It does **not** protect, and is not intended to protect, the much more common case of "a user opens the app normally, dictates something, and never visits Settings → Privacy & Data" — in that case the renderer's normal startup sync (Requirement 5) will complete quickly and push its own default-when-unset value of `0` to main, and from that point on every tick correctly (per the confirmed, deliberate semantics) deletes all audio. This is the intended, confirmed behavior for that case, not a gap — see Problem section.
- Extract the "given a retention value, is it valid and what should happen" decision into a small pure module, `src/helpers/audioCleanupPolicy.js`, exporting:
  - `decideAudioCleanup(retentionDays)` — takes a raw `retentionDays` number and returns whether the cleanup pass should proceed and with what normalized value — valid non-negative finite numbers (including `0`) proceed as-is; invalid values (negative/non-finite) do not.
  - `shouldRunImmediateCleanup(hasBeenSetOnMain)` — the startup-ordering safeguard's skip/proceed decision, extracted as a pure boolean function so it's unit-testable independent of `IPCHandlers`/`main.js` boot wiring.
  - `_setupAudioCleanup()`'s `runCleanup()`/boot logic delegates to both. This mirrors the existing `src/helpers/dictationRouting.js` pattern (pure decision functions extracted specifically so orchestration logic is unit-testable via plain `node --test`).
- Add three new IPC handlers near `get-activation-mode`/`save-activation-mode`/`get-ui-language`/`save-ui-language` (~`ipcHandlers.js:3586-3607`):
  - `ipcMain.handle("get-audio-retention-days", ...)` → returns `this.environmentManager.getAudioRetentionDays()`.
  - `ipcMain.handle("save-audio-retention-days", ...)` → calls and returns `this.environmentManager.saveAudioRetentionDays(days)`.
  - `ipcMain.handle("get-audio-retention-sync-state", ...)` → returns `{ hasBeenSet: environmentManager.hasAudioRetentionDaysBeenSet(), days: environmentManager.getAudioRetentionDays() }`. This is what the renderer's startup sync actually calls (see below) — it needs both pieces of information to decide pull vs. push, which the plain `get-audio-retention-days` getter alone cannot distinguish (it always returns a number, fallback or not).
  - Use `invoke`/`handle` (request/response), matching the `get-ui-language`/`save-ui-language` precedent — no in-memory cache elsewhere needs synchronous updating the way activation mode does.

### `preload.js`

- Expose `getAudioRetentionDays`, `saveAudioRetentionDays`, and `getAudioRetentionSyncState` (each a thin `ipcRenderer.invoke(...)` wrapper) on `window.electronAPI`, alongside `getUiLanguage`/`saveUiLanguage` (~`preload.js:397-398`).

### `src/types/electron.ts`

- Add matching type signatures for `getAudioRetentionDays`, `saveAudioRetentionDays`, and `getAudioRetentionSyncState`, next to `getUiLanguage`/`saveUiLanguage` (~lines 753-755).

### `src/helpers/audioRetentionSync.js` (new)

- Pure decision function, `resolveAudioRetentionStartupSync({ hasBeenSetOnMain, mainValue, rendererValue })`, mirroring the `dictationRouting.js`/`audioCleanupPolicy.js` pattern: if `hasBeenSetOnMain`, return `{ action: "pull", value: mainValue }` (main is authoritative once a real value has genuinely been persisted); otherwise return `{ action: "push", value: rendererValue }` (nothing has ever been persisted, so the renderer's own current value — whether a real prior preference or its own `0` default — must be established as the persisted value, not overwritten by main's fallback). This is the piece of logic Issue 1's fix and its regression test hinge on.

### `src/stores/settingsStore.ts`

- `setAudioRetentionDays(days)` (currently only `localStorage.setItem` + Zustand `set`, lines ~1561-1564): additionally fire-and-forget call `window.electronAPI?.saveAudioRetentionDays?.(days)`, with `.catch()` logging a warning via the existing `logger.warn` pattern used elsewhere in this file for failed syncs.
- Add a startup hydration block alongside the existing `getActivationMode`/`getUiLanguage`/`getVoiceAgentKey`/`getAgentKey` sync-on-load sequence (~lines 2266-2310): call `window.electronAPI.getAudioRetentionSyncState?.()`, feed the result plus the renderer's current `audioRetentionDays` state into `resolveAudioRetentionStartupSync()`, and:
  - `action: "pull"` → if `value` differs from the renderer's current state, update both `localStorage` and Zustand state to match (same shape as the neighboring blocks).
  - `action: "push"` → call `window.electronAPI.saveAudioRetentionDays?.(value)` to establish the renderer's own value as the persisted one on main (no local state change needed — it's already the renderer's current value).
  - try/catch + `logger.warn` on failure, matching the neighboring blocks.
- No change needed to `NUMERIC_SETTINGS`, the cross-window `storage` event listener, or the `0`-default read logic (lines 1065-1071).

### `src/helpers/audioStorage.js` / `src/helpers/meetingAudioStorage.js`

- Add a guard at the top of `cleanupExpiredAudio(retentionDays, ...)` (both files): if `retentionDays` is not a finite number `>= 0`, return immediately with `{ deleted: 0, kept: <count of existing .webm files> }` without touching any file, logging that cleanup was skipped due to an invalid retention value. **Do not** special-case `retentionDays === 0` — let it flow through the existing cutoff computation unchanged, since that formula already correctly deletes everything for `0` (see Problem section). The guard's condition is purely a validity check (`Number.isFinite(retentionDays) && retentionDays >= 0`), not a "is retention disabled" check.

Exact wording left to implementation, but must not contradict the new behavior.
