# Local LLM VRAM/latency tuning: 4k context default + quantized KV-cache + `--fit on` + adaptive context growth

## Status

Implemented

## Addendum (post-implementation production fix)

Live-testing this spec's implementation (commit `d3ed9e16`) surfaced a real bug and prompted two follow-up parameter changes, made directly (not through a separate spec, given their small/mechanical scope and the urgency of a live quality regression):

- **Bug found**: `prewarmServer()` in `modelManagerBridge.js` launched llama-server directly at the model's full registry `contextLength` (e.g. `65536` for Qwen2.5-7B-Instruct) instead of the clamped starting default this spec's design intended — it also never populated `currentContextSizeByModel`, so the first overflow retry would restart the already-large-context server *downward* rather than continuing to grow it. **Fixed**: `prewarmServer()` now computes `Math.min(modelInfo.model.contextLength || DEFAULT_CONTEXT_CAP, DEFAULT_CONTEXT_CAP)` (the same clamp `runInference()` already used) and records the result in `currentContextSizeByModel`. The `ipcHandlers.js` GPU-device-change restart path was similarly fixed to prefer a tracked context size over the raw registry `contextLength`.
- **`DEFAULT_CONTEXT_CAP` lowered from `4096` to `2048`.** Every mention of "4096 is the starting default" and the doubling sequence throughout this document (below) should now be read as **2048** (sequence: `2048→4096→8192→16384→32768→65536`, one extra doubling step).
- **KV-cache quantization changed from `q4_0` to `q8_0`** (`--cache-type-k q8_0 --cache-type-v q8_0`), still gated behind the same binary-capability probe. Rationale: a live dictation-cleanup request (11 characters of input, Qwen2.5-7B-Instruct-Q4_K_M) produced a runaway completion of 2400+ tokens (taking ~1 minute) while running with `q4_0` KV-cache quantization — the more aggressive 4-bit KV-cache, stacked on an already-4-bit-quantized model, is suspected to have degraded output coherence enough that the model failed to emit a stop token promptly. `q8_0` still saves roughly half the f16 KV-cache footprint with much less precision loss.
- Every mention of `q4_0` below reflects the original design intent at the time this spec was written; the shipped, current value is `q8_0`. Tests (`test/helpers/llamaServer.test.js`, `test/helpers/modelManagerBridge.test.js`, `test/helpers/llamaServerGpuRestart.test.js`) and `CLAUDE.md` §8 have been updated to match the current 2048/q8_0 values.

## TL;DR

- Follow-up to `docs/specs/llama-server-ctx-size-fix.md` (Implemented, PR #19). That fix made `--ctx-size` always get _passed_; this spec tunes _what value_ gets passed, adds KV-cache quantization, adds `--fit on` (a real, verified flag on the bundled binary), and makes the context size grow automatically when a request actually needs more room — purely to shrink the local-LLM sidecar's VRAM/RAM footprint/boot time while no longer hard-truncating long chats/notes. Does not touch raw Whisper/Parakeet transcription (separate 500ms budget, CLAUDE.md §3).
- Decisions made:
  - **4096 is now the starting default, not a hard ceiling.** `LlamaServerManager._doStart()` no longer independently enforces 8192; the effective starting value is computed by `modelManagerBridge.js` as `Math.min(modelInfo.model.contextLength || 4096, DEFAULT_CONTEXT_CAP = 4096)` on a fresh (non-retry) start. A new `MAX_CONTEXT_SIZE = 65536` constant remains the real outer cap inside `_doStart` itself, reachable only via the doubling rule below, never chosen directly at first start.
  - **New rule: on a detected "context too small" failure, double and retry, up to 65536.** `llamaServer.js`'s `inference()` now recognizes llama.cpp's context-overflow error signature (HTTP non-200 from `/v1/chat/completions` whose JSON body's error message indicates the prompt/context is too large — matched defensively by keyword pattern, not an exact string, since it can vary by llama.cpp build) and throws a distinguishable `ContextOverflowError`. `modelManagerBridge.js`'s `runInference()` catches that specific error type, restarts the server with `--ctx-size` doubled (4096 → 8192 → 16384 → 32768 → 65536 — one more doubling step than the prior revision, since the starting point dropped from 8192 to 4096), and retries the _same_ original request — not a generic backend-fallback retry (that's the separate, existing GPU-OOM chain in `llamaBackends.js`, untouched).
  - **Model's own registry `contextLength` is still an outer ceiling.** The doubling loop never requests a `--ctx-size` above `Math.min(65536, modelInfo.model.contextLength)` — a model that only declares 2048 will never be pushed to 4096+ even on repeated overflow errors; it simply fails after exhausting that ceiling, same as today.
  - **Doubled size persists for that server process's lifetime, resets on model change/full restart.** Once a restart-and-retry succeeds at a larger size, `modelManagerBridge` remembers that size as the _current_ context for that `currentServerModelId` so subsequent requests to the same model don't immediately overflow again. Switching models (which already triggers a server restart today) resets back to the 4096 default (or the new model's own smaller registry value) — no cross-model persistence, no disk/settings persistence.
  - **Bounded, event-driven retry — not a loop.** At most 4 restart-and-retry attempts per original request (4096→8192, 8192→16384, 16384→32768, 32768→65536), each triggered only by an actual observed overflow failure, never a background timer — compliant with CLAUDE.md §2 idle budget by construction (nothing runs unless a request just failed).
  - **`--fit on` added alongside `--n-gpu-layers 99`, gated by the same `--help`-based capability probe** as KV-cache quantization/flash-attn (one probe function, one `--help` invocation, one cache entry per binary path — see Design). Verified directly against the bundled Windows CPU binary's `--help` output that `-fit, --fit [on|off]` is genuinely supported by this llama.cpp build; other backends (CUDA/Vulkan, downloaded separately) are probed independently since they may be a different build/version.
  - **`--fit`'s interaction with our own explicit `--ctx-size`/`--n-gpu-layers` is a low-risk, best-effort safety net, not a load-bearing mechanism — and this is a residual risk to watch for, not fully verified.** `--fit`'s documented purpose is to adjust _unset_ arguments to fit device memory; we always pass `--ctx-size` and `--n-gpu-layers 99` explicitly, so in the common case `--fit` has little or nothing left to adjust for those two flags specifically. It may still act on other memory-affecting args we don't set (e.g. batch size) — that's the intended benefit. What we could **not** fully verify from available documentation is whether llama.cpp's `--fit` implementation ever second-guesses or silently caps an _explicitly_ set `--ctx-size` when it judges the value won't fit (its `--fit-ctx` default of 4096, i.e. "the minimum ctx size --fit itself will not go below," is a coincidental match with our new starting default, not evidence either way about whether it touches explicit values at all). Because `--fit` is applied unconditionally whenever supported (independent of which `--ctx-size` the doubling logic is currently requesting), an unexpected interaction — e.g. `--fit` capping something the adaptive-doubling retry didn't anticipate — is called out explicitly as something manual validation must watch for, not asserted away.
  - No blocking open question for the parts we could verify (detection signature, retry ceiling, persistence, `--fit` support on the CPU binary, and the interaction with the model's own `contextLength`) — the one flagged unresolved point (`--fit`'s exact interaction with explicit `--ctx-size`) is a documented residual risk with a manual-validation watch-step, not a blocking decision; the user asked for the flag to be added "if supported," which this spec satisfies.
- Practical impact: local-LLM passes now default to a smaller/faster 4096 context, quantized KV-cache, and `--fit on` (where supported) for the common short-prompt case, but a long `chatIntelligence` conversation or a long pasted `noteFormatting` note that previously got silently truncated now triggers one or more automatic (and visibly logged) server restarts to a larger context, up to 65536 or the model's own ceiling, before falling back to today's truncation behavior only if that's still not enough. The user experiences a brief added restart delay (roughly the existing server-boot time, once per doubling step needed) instead of silent truncation, plus a marginally lower baseline VRAM footprint from the smaller starting context.

## Problem / Goal

After `docs/specs/llama-server-ctx-size-fix.md` shipped, `--ctx-size` is now always passed, sourced from the model registry's `contextLength` (e.g. 262144 for `nemotron-3-nano-4b-q4_k_m`). That value is still far larger than needed for this app's actual local-LLM use cases — dictation cleanup, dictation-agent commands, note formatting, and chat intelligence are predominantly short-prompt, low-latency workloads, not long-document/long-conversation workloads that need a huge context window. A large `--ctx-size` means a large KV-cache allocation regardless of how short the actual prompt is, which:

- Wastes VRAM/RAM proportional to `ctx-size`, unrelated to what's actually used.
- Slows server boot (KV-cache buffer allocation time).
- Can still risk a GPU OOM on constrained VRAM configurations even at a "reasonable" registry `contextLength`, if the GPU is small or shared with other workloads.

Additionally, the KV-cache today always uses the default `f16` (2 bytes/element) precision. llama.cpp supports quantizing it to `q4_0` (roughly a quarter of the f16 footprint), which — combined with capping the context itself — compounds the memory savings. The vendored binary also supports `--fit on`, which can adjust other unset memory-affecting launch arguments to fit the device, as an additional (best-effort) safety net.

## Requirements

- `LlamaServerManager._doStart()` must clamp the resolved `--ctx-size` value to a new `MAX_CONTEXT_SIZE = 65536` ceiling, applied as `Math.min(resolved, MAX_CONTEXT_SIZE)`, where `resolved` is exactly the value `_doStart` already computes today (`options.contextSize` if a finite positive number, else the existing `DEFAULT_CONTEXT_SIZE = 4096` fallback). `_doStart` itself doesn't know about doubling or the 4096 starting default — it just launches whatever `--ctx-size` it's asked for, clamped only against the real 65536 ceiling. It is `modelManagerBridge.runInference()`'s job to decide whether to ask for 4096 (first attempt) or a doubled value (retry attempt) — `_doStart` never independently reintroduces a hard cap for a caller that's deliberately requesting more room after an overflow.
- Add a `DEFAULT_CONTEXT_CAP = 4096` constant in `llamaServer.js`, exported the same way `DEFAULT_CONTEXT_SIZE`/`MAX_CONTEXT_SIZE` are — this is the value `modelManagerBridge.js` targets on a fresh (non-retry) server start for a model: `Math.min(modelInfo.model.contextLength || 4096, DEFAULT_CONTEXT_CAP)`. `_doStart`'s own clamp remains `Math.min(resolved, MAX_CONTEXT_SIZE)` (65536) — the practical effect of "4096 by default" comes entirely from what `modelManagerBridge.js` passes as `options.contextSize` on a first start, not from `_doStart` refusing anything above 4096.
- `llamaServer.js`'s `inference()` must distinguish a context-overflow failure from any other non-200 response. On a non-200 response, parse the JSON error body if possible and test its message against a defensive, case-insensitive keyword pattern for llama.cpp's context-overflow phrasing (e.g. requires the word "context" together with one of "exceed", "too long", "too large", or "increase"). If it matches, throw a new exported `ContextOverflowError` (extends `Error`, sets `this.isContextOverflow = true`) instead of the current generic `Error`; otherwise throw the existing generic `Error` unchanged. This must not fire for unrelated 4xx/5xx failures (malformed request, server crash, timeout) — those keep throwing the plain `Error` as today.
- `modelManagerBridge.js`'s `runInference()` must catch `ContextOverflowError` (via `instanceof` or the `isContextOverflow` flag, not string-matching) specifically around its `serverManager.inference()` call, and, if the currently-loaded context size is below `Math.min(MAX_CONTEXT_SIZE, modelInfo.model.contextLength || MAX_CONTEXT_SIZE)`, restart the server for the same `modelId` with `contextSize` doubled (current × 2, capped at that same ceiling) and retry the original `inference()` call once for that new size. This may repeat up to the point the ceiling is reached; if `ContextOverflowError` is thrown again at the ceiling (or the model's own `contextLength` ceiling is already met), the error propagates to the caller unchanged — no infinite loop, no silent success-without-actually-fitting.
- `modelManagerBridge.js` must track the current effective context size per loaded model (e.g. alongside its existing `currentServerModelId` tracking) so that a later request to the same already-larger-context server doesn't have to rediscover the larger size via another overflow round-trip — it reuses the last-successful size for that `modelId` until the model changes or the server otherwise restarts, at which point it resets to the `Math.min(modelInfo.model.contextLength || 4096, DEFAULT_CONTEXT_CAP)` default.
- The doubling loop must never request a `--ctx-size` greater than `Math.min(MAX_CONTEXT_SIZE, modelInfo.model.contextLength)` when the model's registry `contextLength` is a finite positive number smaller than 65536 — the model's own declared capability is always respected as a hard outer bound, independent of the global 65536 cap.
- The doubling retry is strictly bounded: starting from 4096, only 4096→8192→16384→32768→65536 are reachable (at most 4 restart-and-retry attempts for one original request), and it is only ever triggered by an actual `ContextOverflowError` from a real request — never a timer, poll, or speculative pre-emptive restart.
- Before adding KV-cache-quantization/flash-attention/`--fit` flags to `baseArgs`, the manager must probe the actual resolved binary for the specific backend about to be started, via a single `--help` invocation, checking its output for the presence of `--cache-type-k`, `--cache-type-v`, `--flash-attn`, and (independently) `-fit`/`--fit`. Only add `--cache-type-k q4_0`, `--cache-type-v q4_0`, `--flash-attn on` if all three of those are present; independently, only add `--fit on` (placed alongside `--n-gpu-layers 99` in `baseArgs`) if `-fit`/`--fit` is present — the two feature sets are gated independently off the same single probe result, since a given binary could in principle support one without the other. This probe (one `--help` invocation, one cache entry per binary path) must be cached per binary path (mirroring `llamaBackends.js`'s existing `vulkanDeviceCache` pattern) so it only runs once per binary per app run, not on every server start/restart, and not as two separate execs for the two feature sets.
- If a backend's start attempt with the quantization/flash-attn/`--fit` flags present fails (process dies during startup, non-zero/`SIGKILL` exit, or startup timeout — the same failure surface `_doStart`'s existing per-backend retry loop already detects), retry starting that same backend exactly once with all of those flags stripped (falling back to default f16 cache, no explicit flash-attn override, no `--fit`) before moving on to the next backend in the chain. This must not increase the total number of backends tried beyond "each backend, optionally twice (with-flags then without)" — it must not turn into an unbounded retry loop.
- The GPU-device-change restart path in `ipcHandlers.js` (`purpose === "intelligence"` branch, already fixed by the prior spec to preserve `contextSize` across restart) needs **no further change** — since the 4096 cap and the KV-cache/flash-attn/`--fit` flags are both applied centrally inside `_doStart` regardless of what `options.contextSize` the caller passes, that path automatically inherits both the cap and the quantization/`--fit` behavior without modification.
- No functional change to `--n-gpu-layers`'s value (`99`), `--jinja`, `--threads`, `--host`, `--port`, backend selection/ordering, or the existing per-backend startup-failure retry semantics beyond the one additional flags-stripped retry described above. `--fit on` is additive alongside `--n-gpu-layers 99`, not a replacement for it.
- Must not alter per-scope prompt-building, truncation, or the four-scope config in `src/config/inferenceScopes.ts` — this is scoped entirely to the llama-server process launch arguments.

## Non-goals

- Not implementing per-scope context sizes or multiple concurrent llama-server processes. The single shared `LlamaServerManager` remains a singleton with one active context size at a time, shared by every scope currently routed to it. A future spec could revisit this if the doubling behavior here still proves insufficient in practice.
- Not changing the model registry's declared `contextLength` values — they remain correct metadata about each model's native capability; both the 4096 default and the 65536 ceiling are launch-time values layered on top of whatever the registry says, never exceeding it.
- Not adding a user-facing settings toggle for context size, KV-cache quantization, or the doubling ceiling — all remain internal tuning constants, consistent with how `contextLength` itself has no UI control today.
- Not changing `llamaBackends.js`'s backend selection, device-pinning, or GPU-layer offload logic — the doubling retry is orthogonal to (and layered on top of) that existing per-backend GPU-fallback chain, not a replacement for it.
- Not persisting the doubled context size anywhere outside in-memory process state (no settings key, no database row, no `.env` entry) — it resets on model change or app restart by design (see TL;DR).
- Not addressing the (separately tracked, unconfirmed) "garbled output" observation from the prior spec.
- Not touching whisper.cpp/Parakeet transcription paths.

## Design

### `src/helpers/llamaServer.js`

Add new constants alongside the existing `DEFAULT_CONTEXT_SIZE`:

- `DEFAULT_CONTEXT_CAP = 4096` — the value `modelManagerBridge.js` targets on a fresh (non-retry) server start for a model, exported the same way `DEFAULT_CONTEXT_SIZE` already is. (Revised down from an earlier 8192 draft value — see TL;DR.)
- `MAX_CONTEXT_SIZE = 65536` — the real outer ceiling `_doStart` itself enforces. `_doStart` has no concept of "default vs. doubled" — it just launches whatever `--ctx-size` it's given, clamped only against this 65536 ceiling. The 4096-by-default behavior for ordinary (non-overflow) requests comes entirely from what `modelManagerBridge.js` chooses to pass in, not from `_doStart` refusing larger values.

In `_doStart(modelPath, options = {})`, where the `--ctx-size` value is currently computed inline in the `baseArgs` array literal, change the computation to first resolve the existing value (unchanged logic: `options.contextSize` if finite and positive, else `DEFAULT_CONTEXT_SIZE`), then clamp it with `Math.min(resolved, MAX_CONTEXT_SIZE)` before stringifying it into `baseArgs`. Store the clamped numeric value in a local variable (e.g. `ctxSize`) since it's also useful for logging.

**Context-overflow detection in `inference()`.** In the existing `res.on("end", ...)` handler, when `res.statusCode !== 200`, attempt `JSON.parse(data)` and read `parsed?.error?.message` (llama.cpp's OpenAI-compatible error shape) if present. Test that message (case-insensitively) against a new exported helper `isContextOverflowMessage(message)`: returns `true` only if the message contains "context" together with at least one of "exceed", "too long", "too large", or "increase" (all substring checks, not exact phrases, to tolerate wording differences across llama.cpp builds/tags). If it matches, reject with a new exported `ContextOverflowError` class (`class ContextOverflowError extends Error { constructor(message) { super(message); this.name = "ContextOverflowError"; this.isContextOverflow = true; } }`) carrying the original server message; otherwise keep today's behavior (`reject(new Error(...))`) unchanged for every other non-200 case, including JSON-parse failures on the error body (treated as a non-overflow error, since the shape couldn't be confirmed).

Add a single binary-capability probe, analogous in spirit to `llamaBackends.js`'s `listVulkanDevices`/`vulkanDeviceCache`: a module-level cache (`Map<binaryPath, {kvCacheQuant: boolean, fit: boolean}>`) and a function `probeBinaryCapabilities(binaryPath)` that runs the binary with `--help` once (via `execFileSync`, short timeout, `windowsHide: true`, matching the existing `listVulkanDevices` invocation style) and returns `{ kvCacheQuant, fit }`, where `kvCacheQuant` is `true` only if the captured stdout contains all of `--cache-type-k`, `--cache-type-v`, and `--flash-attn`, and `fit` is `true` only if the stdout contains `-fit` or `--fit` (the two capabilities are independent booleans derived from the same single `--help` output, not two separate execs). On any error (spawn failure, non-zero exit, timeout), return `{ kvCacheQuant: false, fit: false }` and cache that outcome too, so a broken/foreign binary is only probed once, not on every start attempt. This function belongs in `llamaServer.js` (not `llamaBackends.js`) since it's about `_doStart`'s own launch-arg construction, though it takes a `binaryPath` argument the same way `listVulkanDevices` does, so it can be tested independent of any specific backend. (Named `probeBinaryCapabilities` rather than the earlier draft's `supportsKvCacheQuantization` now that it covers two independent flag families off one probe.)

In `_startBackend(backend, baseArgs, gpuMode)` (or a thin wrapper around it), before calling `backend.buildArgs(baseArgs, gpuMode)`, resolve `backend.getBinaryPath()` (already computed inside `buildArgs`/`getBinaryPath` per-backend) and call `probeBinaryCapabilities(binaryPath)`. If `kvCacheQuant` is `true`, append `--cache-type-k`, `q4_0`, `--cache-type-v`, `q4_0`, `--flash-attn`, `on` to the args passed to that backend's launch. Independently, if `fit` is `true`, append `--fit`, `on` immediately alongside the existing `--n-gpu-layers`, `99` pair already present in `baseArgs` (same position in the arg list, so the two GPU/device-memory-related flags stay adjacent in logs and in the actual argv). Order among flags doesn't matter to llama-server itself; this is purely for log/argv readability. If a capability is `false`, launch without its corresponding flags (today's default f16 cache / `auto` flash-attn / no `--fit` behavior, preserved exactly).

**Whether `--fit on` does anything useful here, given `--ctx-size` and `--n-gpu-layers` are both explicitly set:** per llama.cpp's own `--help` text, `--fit [on|off]` adjusts _unset_ arguments to fit device memory; `--ctx-size` and `--n-gpu-layers` are always explicitly set by this codebase, so for those two specific flags `--fit` should have nothing to override in the common case — its practical effect here is expected to be on _other_ memory-affecting arguments this codebase never sets explicitly (e.g. batch size, `--ubatch-size`, or similar), where it can still help avoid an OOM on a constrained device. This is treated as a **low-risk, complementary safety net addition**, not a load-bearing replacement for the doubling/clamping logic already in this spec. **Residual, unverified risk**: llama.cpp's public documentation does not fully specify whether `--fit`'s memory-fitting pass ever inspects or silently second-guesses an _explicitly_-set `--ctx-size` (e.g., downsizing it if it judges the value won't fit alongside `--n-gpu-layers 99`), and this could not be confirmed against source without deeper llama.cpp-internals investigation out of scope for this spec. `--fit-ctx`'s documented default of 4096 ("minimum ctx size that can be set by `--fit`") coincidentally matches this spec's new `DEFAULT_CONTEXT_CAP`, but that is noted as a coincidence, not evidence about whether `--fit` touches our explicit value at all. `--fit` is applied unconditionally whenever supported, independent of which `--ctx-size` the adaptive-doubling retry is currently requesting — the two mechanisms are orthogonal by design (see Requirements), but if `--fit` were ever observed silently capping or altering the effective context size in a way that conflicts with the doubling retry's expectations, that would be a bug to address in a follow-up, not something this spec can rule out today. The Validation Plan below adds a manual check to watch for exactly this.

Extend `_doStart`'s existing per-backend retry loop (the `for (const backend of chain)` loop that already catches a failed `_startBackend` and moves to the next backend) with one additional retry tier: when a backend's _first_ attempt included the KV-cache-quantization/flash-attn/`--fit` flags and that attempt fails, retry the same backend once more with all of those flags omitted (plain f16, no explicit flash-attn, no `--fit`) before falling through to the next backend in `chain`. Track this per backend with a simple flag/counter local to the loop iteration — this must not multiply total attempts beyond "each backend, at most twice." If a backend's binary didn't advertise support for either capability in the first place (probe returned `false` for both), there is only ever one attempt for it, same as today.

The existing `debugLogger.info("llama-server launch parameters", ...)` call must include whether KV-cache quantization and `--fit` were applied for that attempt (e.g. added `kvCacheQuantized: true/false` and `fitEnabled: true/false` fields) so both are visible in debug logs the same way the rest of the launch config already is.

No changes to `_killCurrentProcess`, `checkHealth`, `startHealthCheck`, `stop`, `getStatus`, `resetGpuDetection`, or any code outside `_doStart`/`_startBackend`'s immediate surroundings.

### `src/helpers/modelManagerBridge.js` and `src/helpers/ipcHandlers.js`

`prewarmServer()`, the GPU-restart path (`ipcHandlers.js`, `purpose === "intelligence"`), and every other existing call site keep passing `options.contextSize` derived from the registry's `contextLength` straight through to `serverManager.start()` — the `MAX_CONTEXT_SIZE = 65536` clamp and the KV-cache/flash-attn/`--fit` logic all still live centrally in `_doStart`, so those call sites inherit that behavior automatically without modification, same as the prior design. No change needed there.

`runInference()` is the one call site that changes, since it's the only place both an actual inference request and the resulting error are observed (per the analysis above: this is deliberately _not_ placed in `_doStart`'s own per-backend chain, because that chain is about which GPU/CPU backend to use for a fixed context size, while this new rule is about the context size itself being insufficient for a given request's content, independent of backend):

- Add an in-memory `Map<modelId, currentContextSize>` (e.g. `this.currentContextSizeByModel`) on the bridge instance, alongside the existing `this.currentServerModelId` tracking. Not persisted to disk/settings — resets whenever the process restarts (app relaunch) or whenever `currentServerModelId` changes to a different model (the existing "Start/restart server if needed or if model changed" branch already forces a fresh `serverManager.start()` in that case, which now also resets the tracked size for the new model back to its own default).
- On a fresh (non-retry) start for `modelId`, the target `contextSize` passed to `serverManager.start()` becomes `Math.min(modelInfo.model.contextLength || DEFAULT_CONTEXT_CAP, DEFAULT_CONTEXT_CAP)` where `DEFAULT_CONTEXT_CAP = 4096` — i.e. the existing `options.contextSize || modelInfo.model.contextLength || 4096` line is replaced with a computation that additionally clamps to `DEFAULT_CONTEXT_CAP` (imported from `llamaServer.js`) for a first start, and the resulting value is recorded in `currentContextSizeByModel`.
- Wrap the existing `this.serverManager.inference(messages, {...})` call (inside `runInference()`'s existing `try`) in a small retry loop: on `ContextOverflowError` (checked via `instanceof` import from `llamaServer.js`, or the `isContextOverflow` flag as a fallback if instance identity is ever a concern across module boundaries), compute `nextSize = Math.min(currentSize * 2, MAX_CONTEXT_SIZE, modelInfo.model.contextLength || MAX_CONTEXT_SIZE)`. If `nextSize > currentSize`, call `this.serverManager.start(modelPath, { contextSize: nextSize, threads: ..., gpuLayers: 99 })` again (same model, same path — this is a restart of the same process with a larger `--ctx-size`, not a different backend), update `currentContextSizeByModel.set(modelId, nextSize)`, and retry the same `serverManager.inference()` call with the same `messages`/options once. If `nextSize` is not greater than `currentSize` (ceiling already reached, either the 65536 global cap or the model's own smaller `contextLength`), do not retry — let the `ContextOverflowError` propagate to the caller as-is, preserving today's "long input gets an error/truncation" outcome as the final fallback once genuinely out of room.
- Log every doubling attempt via `debugLogger.logReasoning("INFERENCE_CONTEXT_OVERFLOW_RETRY", { modelId, fromSize: currentSize, toSize: nextSize })` so the behavior is visible in debug logs the same way the rest of the inference pipeline already is.
- `ipcHandlers.js`'s GPU-restart path needs no change — it already preserves the currently-tracked `contextSize` across a device-switch restart per the prior spec, and will simply pick up whatever `currentContextSizeByModel` currently holds when it next queries the bridge for the active context, since that tracking lives on the same bridge instance.

### Compliance with Non-Negotiable Product Premises

- **Speed (§3)**: affects only the optional, opt-in local-LLM cleanup/agent/note/chat pass, which has its own latency budget separate from the 500ms raw-transcription figure (CLAUDE.md §3) — no change to the Whisper/Parakeet pipeline. Expected effect on this separate budget is net-positive or neutral for the common case: a smaller default `--ctx-size` (4096) and quantized KV-cache reduce the memory the server must allocate at boot (faster boot, especially for large-native-context models) and reduce memory bandwidth pressure during inference; `--fit on` (where supported) is an additional best-effort safety net against OOM on constrained devices for other unset memory-affecting args (see Design's residual-risk discussion). The new doubling-retry rule trades a bounded, occasional extra restart delay (only when a request genuinely overflows the current size, at most 4 restarts to reach 65536 from the 4096 starting point) for avoiding silent truncation — this is a deliberate, visible latency cost on an already-opt-in, already-separately-budgeted pass, not a regression against the 500ms transcription figure.
- **Idle budget (§2)**: no new always-on timer or polling loop. The binary-capability probe (`--help` via `execFileSync`, covering both KV-cache/flash-attn and `--fit` support in one invocation) runs at most once per binary path per app run (cached), triggered only when a server is actually starting — the same lazy-spawn-triggered, cached-check pattern already used and accepted for `listVulkanDevices`/`vulkanDeviceCache`. The context-doubling restart-and-retry is likewise strictly event-driven: it only executes in direct response to an observed `ContextOverflowError` from a real inference request just made by the user, never on a timer, poll, or speculative background trigger, and is bounded to at most 4 restarts per request — no idle-time cost, no background work while the app is otherwise idle.
- **Privacy (§1)**: no new network calls, no new listener/port. `--ctx-size`, `--cache-type-k/-v`, `--flash-attn`, `--fit` are local process launch arguments only.
- **Migration safety (§6)**: no settings/schema/storage format changes — `DEFAULT_CONTEXT_CAP`, `MAX_CONTEXT_SIZE`, the KV-cache/`--fit` flags, and the in-memory `currentContextSizeByModel` map are internal constants/process-lifetime state, not persisted user data.
- **Graceful degradation (§5)**: this is the central design concern of this spec — see the capability probe (never assume flag support, always verify against the actual binary's own `--help`, independently for KV-cache/flash-attn vs. `--fit`) and the flags-stripped retry-once-per-backend fallback above. A binary that doesn't support these flags, or that crashes when given them despite advertising support, degrades to today's known-working f16/no-flash-attn/no-`--fit` behavior for that backend rather than breaking the local-LLM feature.

### Docs

- `CLAUDE.md`'s Model Registry Architecture section (§8) should gain a one-line note that a model's registry `contextLength` is treated as a _maximum request_: the local server starts at a 4096 default per request, doubles automatically (4096→8192→16384→32768→65536) on a detected context-overflow failure, and never exceeds the smaller of 65536 or the model's own declared `contextLength` — so a future reader doesn't assume registry `contextLength` is what's actually always used at runtime, nor that 4096 is a hard ceiling. The note should also mention `--fit on` is added when the resolved binary supports it, alongside `--n-gpu-layers 99`, as a best-effort memory-fitting safety net independent of the doubling logic.

## Open Questions

None blocking. The per-scope context-size ambiguity raised in the original task is resolved architecturally (single shared server process means a global default/cap is the only option today; see TL;DR and Non-goals). The context-doubling rule resolves the accepted-tradeoff truncation concern for `chatIntelligence`/`noteFormatting` directly, rather than deferring it to a future spec — detection signature, retry ceiling (65536), starting default (4096), interaction with the model's own `contextLength`, and persistence semantics (process-lifetime only, resets on model change/restart) are all decided above. If the doubling ceiling of 65536 itself still proves insufficient for some future extreme-length use case, that would warrant a follow-up spec, not a revision of this one. The one non-blocking residual risk — whether llama.cpp's `--fit on` implementation ever adjusts an explicitly-set `--ctx-size` in a way that could interact unexpectedly with the doubling logic — is documented in Design and given a dedicated manual-validation watch-step; it is not blocking approval since `--fit` is additive/best-effort and degrades gracefully if unsupported, but should be watched in practice per that validation step.
